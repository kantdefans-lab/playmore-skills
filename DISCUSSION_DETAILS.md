# 动态概念库系统 - 细节讨论清单

## 1️⃣ 数据结构细节

### 1.1 概念ID的生成和管理

**当前设计**：
- 种子概念：`seed_001`, `seed_002`, ...
- 自动概念：`auto_001`, `auto_002`, ...

**可讨论的问题**：
- [ ] ID是否应该基于概念名称的hash？（便于跨session一致性）
  - 优点：即使重启也能识别同一概念
  - 缺点：hash冲突风险，难以调试
  
- [ ] 是否需要版本号？（如`auto_001_v1`, `auto_001_v2`）
  - 优点：追踪概念的演进
  - 缺点：增加复杂度
  
- [ ] 如何处理用户手动创建的概念？（不是种子，也不是自动识别）
  - 选项A：用`manual_001`前缀
  - 选项B：混入`auto_`序列
  - 选项C：单独的`user_001`前缀

**建议**：保持简单，用`seed_/auto_/manual_`三个前缀

---

### 1.2 概念的元数据字段

**当前设计**包含：
- id, name, description, keywords
- parent_concept_id, confidence, status
- created_at, last_updated, paper_count

**可讨论的问题**：
- [ ] 是否需要`aliases`字段？（同义词）
  ```json
  {
    "name": "Transformer模型",
    "aliases": ["Transformer", "Transformer Architecture", "Attention is All You Need"]
  }
  ```
  - 用途：改进概念匹配准确度
  - 成本：增加维护负担

- [ ] 是否需要`tags`字段？（分类标签）
  ```json
  {
    "name": "Transformer模型",
    "tags": ["architecture", "deep-learning", "nlp"]
  }
  ```
  - 用途：便于按类型过滤
  - 成本：需要定义tag体系

- [ ] 是否需要`source_url`字段？（概念来源）
  ```json
  {
    "name": "Transformer模型",
    "source_url": "https://arxiv.org/abs/1706.03762"
  }
  ```
  - 用途：追踪概念的原始论文
  - 成本：需要手动或自动填充

- [ ] 是否需要`user_notes`字段？（用户备注）
  ```json
  {
    "name": "Transformer模型",
    "user_notes": "这个概念在我们的研究中很重要"
  }
  ```

**建议**：先不加aliases和tags，保持简洁。source_url和user_notes可选。

---

### 1.3 概念关系的表示

**当前设计**：
```json
{
  "source_id": "seed_001",
  "target_id": "auto_001",
  "relation_type": "contains",
  "strength": 0.95
}
```

**可讨论的问题**：
- [ ] 关系类型是否足够？
  - 当前：`contains`, `related_to`, `similar_to`
  - 是否需要：`uses`, `implements`, `extends`, `conflicts_with`?
  
- [ ] 是否需要双向关系？
  - 当前：单向（source → target）
  - 是否应该自动生成反向关系？
  - 例如：`contains` 的反向是 `contained_by`

- [ ] `strength`字段的含义是什么？
  - 当前：0-1的相关度
  - 是否应该用百分比或其他单位？

**建议**：
- 关系类型保持简单：`parent_of`, `related_to`, `similar_to`
- 不需要双向关系，用户查询时可以反向查询
- strength保持0-1，便于计算

## 2️⃣ Claude Prompt设计细节

### 2.1 概念识别的Prompt

**当前设计**：给Claude一个论文列表，让它识别概念

**可讨论的问题**：
- [ ] 是否应该一次处理多篇论文，还是逐篇处理？
  - 一次多篇：
    - 优点：token效率高，可以识别跨论文的概念关系
    - 缺点：context可能过长，容易遗漏细节
  - 逐篇处理：
    - 优点：精准度高，易于调试
    - 缺点：token成本高，无法识别跨论文关系

- [ ] Prompt中是否应该包含已有概念的详细信息？
  - 当前：只包含概念名称和关键词
  - 是否应该包含：概念描述、相关论文、已有笔记？
  - 成本：增加token消耗

- [ ] 是否应该给Claude一个"概念提取指南"？
  - 例如：什么样的词汇应该成为概念？
  - 例如：排除通用词、人名、地名等

- [ ] 是否应该让Claude同时识别概念关系？
  - 当前：只识别概念本身
  - 是否应该让Claude识别：parent/child/related关系？

**建议**：
- 逐篇处理（精准度优先）
- 包含已有概念的基本信息（名称+关键词+描述）
- 提供"概念提取指南"（什么是好的概念）
- 同时识别概念关系

### 2.2 置信度计算的Prompt

**当前设计**：Claude直接输出0-1的置信度

**可讨论的问题**：
- [ ] 置信度应该基于什么因素？
  - 当前：Claude的主观评估
  - 是否应该包含：
    - 关键词匹配度？
    - 在多篇论文中出现的频率？
    - 与种子概念的距离？

- [ ] 是否应该让Claude解释置信度的原因？
  ```json
  {
    "name": "Prompt Engineering",
    "confidence": 0.85,
    "confidence_reasons": [
      "在论文标题中出现",
      "在摘要中出现2次",
      "与种子概念'AI×心理健康'高度相关"
    ]
  }
  ```

- [ ] 是否应该有多个置信度维度？
  ```json
  {
    "name": "Prompt Engineering",
    "confidence": {
      "relevance": 0.85,  // 与种子概念的相关度
      "clarity": 0.90,    // 概念定义的清晰度
      "importance": 0.75  // 在论文中的重要性
    }
  }
  ```

**建议**：
- 置信度基于多个因素（关键词+频率+相关度）
- 让Claude解释原因（便于用户理解）
- 暂不使用多维置信度（保持简洁）

### 2.3 层级关系识别的Prompt

**当前设计**：单独的/playmore-organize-concepts命令

**可讨论的问题**：
- [ ] 是否应该在概念识别时就识别层级关系？
  - 当前：两个独立的步骤
  - 是否应该合并？
  - 成本：增加单次Prompt的复杂度

- [ ] 层级关系的识别规则是什么？
  - 是否应该基于概念名称的结构？
  - 是否应该基于语义相似度？
  - 是否应该基于论文中的共现频率？

- [ ] 是否应该允许多个parent？
  - 当前：单个parent_concept_id
  - 是否应该支持多继承？
  - 例如：Prompt Engineering既属于AI×心理健康，也属于NLP技术

**建议**：
- 保持两个独立步骤（概念识别 + 层级整理）
- 层级关系基于语义相似度和共现频率
- 暂不支持多继承（保持简洁）

## 3️⃣ 概念识别算法细节

### 3.1 关键词匹配的方式

**当前设计**：简单的字符串匹配

**可讨论的问题**：
- [ ] 是否应该使用模糊匹配？
  - 例如：`Transformer` 匹配 `Transformers`
  - 工具：Levenshtein距离、Jaro-Winkler等

- [ ] 是否应该使用语义匹配？
  - 例如：`LLM` 匹配 `Large Language Model`
  - 工具：词向量、embedding等
  - 成本：需要额外的模型或API

- [ ] 是否应该考虑词序？
  - 例如：`Prompt Engineering` vs `Engineering Prompt`

- [ ] 是否应该支持正则表达式？
  - 例如：`Transformer.*` 匹配所有Transformer相关的概念

**建议**：
- 使用模糊匹配（Levenshtein距离 > 0.8）
- 暂不使用语义匹配（成本高）
- 不考虑词序（简化处理）
- 不支持正则表达式（保持简洁）

**当前设计**：用户手动合并

**可讨论的问题**：

  - 例如：`Prompt Engineering` 和 `Prompt Design` 是否应该合并？
  - 阈值：相似度 > 0.8 时提示用户

- [ ] 合并时应该保留哪个名称？
  - 选项A：保留第一个

  - 选项C：让用户选择

- [ ] 合并时应该如何处理关键词？
  - 选项A：合并所有关键词
  - 选项B：去重后合并
  - 选项C：让用户选择

- [ ] 合并时应该如何处理论文映射？
  - 是否应该自动更新paper_concepts.json？
  - 是否应该保留合并历史？

**建议**：
- 自动检测重复（相似度 > 0.85）

## 4️⃣ 置信度阈值细节

### 4.1 自动批准的阈值

**当前设计**：confidence > 0.9 自动批准

**可讨论的问题**：
- [ ] 0.9是否是最优阈值？
  - 太高（0.95+）：很少有概念自动批准，用户工作量大
  - 太低（0.7-）：容易批准低质量概念，需要后续清理
  - 建议：0.85-0.90之间

- [ ] 是否应该根据概念类型调整阈值？
  - 例如：技术概念要求0.9，应用概念要求0.8
  - 成本：增加复杂度

- [ ] 是否应该根据论文数量调整阈值？
  - 例如：在多篇论文中出现的概念，阈值可以降低
  - 逻辑：频繁出现 = 更可靠

- [ ] 是否应该有"软阈值"和"硬阈值"？
  - 软阈值（0.85）：自动批准，但标记为"需要验证"
  - 硬阈值（0.95）：完全自动批准

**建议**：
- 自动批准阈值：0.85
- 不根据概念类型调整（保持简洁）
- 根据论文频率调整：在3+篇论文中出现，阈值降低0.05
- 不使用软硬阈值（保持简洁）

---

### 4.2 拒绝的处理

**当前设计**：用户拒绝后，标记为rejected

**可讨论的问题**：
- [ ] 拒绝的概念是否应该再次出现？
  - 当前：一旦拒绝，永久拒绝
  - 是否应该允许"重新考虑"？
  - 例如：如果在5+篇论文中出现，自动重新考虑

- [ ] 是否应该记录拒绝的原因？
  ```json
  {
    "status": "rejected",
    "rejection_reason": "与现有概念重复",
    "rejected_at": "2026-04-27",
    "rejected_by": "user"
  }
  ```

- [ ] 是否应该有"拒绝"的等级？
  - 例如：`soft_reject`（可能重新考虑）vs `hard_reject`（永久拒绝）

**建议**：
- 拒绝的概念可以重新考虑（如果在5+篇论文中出现）
- 记录拒绝原因和时间
- 不使用拒绝等级（保持简洁）
